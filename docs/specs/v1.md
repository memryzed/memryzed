# Memryzed v1 specification

Status: draft, source of truth for the v1 implementation.
Owner: project maintainers.
Last updated: alongside this document set.

This is the specification for the v1.0 release of Memryzed. It is the
authoritative reference for what is being built. When user-facing
documentation and this specification disagree, this document wins
until v1.0 ships, after which `CHANGELOG.md` and the user
documentation become authoritative.

## 1. Goals

Memryzed v1 must do these things well:

1. Give any MCP-aware coding agent a stable, durable memory store.
2. Let users resume the exact task they were working on in any
   repository.
3. Run entirely on the user's machine, with no required network
   access after install.
4. Install in under 30 seconds with a single command, on every major
   developer platform.
5. Wire itself into the user's existing MCP clients automatically,
   so the user does not edit JSON.
6. Keep the user in control of what is remembered through a visible,
   editable trust loop.

These goals drive every other decision in this spec.

## 2. Non-goals

Things v1 does not do:

- Cloud sync between machines.
- Team or organization features.
- A web user interface.
- Authentication or accounts.
- Encryption of data at rest.
- A separate HTTP API outside of MCP.
- Telemetry beyond opt-in counters.
- Custom embedding models beyond the bundled BGE-small.
- Crawling or auto-indexing the user's filesystem.

These are deferred either to later v1.x patches or to the future
paid product.

## 3. Personas

The product is designed for these users:

- The individual developer who uses one or more AI coding agents
  daily, runs into the "agent forgot everything" problem, and wants
  a fix that works across whichever agent they prefer this month.
- The agent author who wants to give their users persistent memory
  without building it themselves, and is willing to integrate with a
  small, stable MCP surface.

A future tier targets teams. v1 does not.

## 4. Brand and identity

- Name: Memryzed.
- Wordmark: capitalized "Memryzed" in prose. CLI binary, package
  names, and module names are lowercase `memryzed`.
- Domain: `memryzed.com` (primary). `memryzed.io` redirects to it.
- Repository: `github.com/memryzed/memryzed`.
- License: Apache-2.0.
- Tone: serious, useful, professional. Memryzed helps developers do

## 5. Distribution

### 5.1 Install commands

Every supported platform installs through one of these one-line
commands:

    macOS, Linux, WSL:    curl -fsSL https://memryzed.com/install.sh | bash
    Windows PowerShell:   irm https://memryzed.com/install.ps1 | iex
    Windows CMD:          curl -fsSL https://memryzed.com/install.cmd -o install.cmd && install.cmd

The install scripts:

1. Detect operating system and CPU architecture.
2. Download the matching release binary from GitHub Releases.
3. Verify its SHA-256 checksum against a manifest signed by the
   project's release key.
4. Place the binary at `~/.memryzed/bin/memryzed` (Unix) or
   `%LOCALAPPDATA%\memryzed\bin\memryzed.exe` (Windows).
5. Add the binary's directory to the user's PATH.
6. Print three numbered next steps.

### 5.2 Build matrix

The release pipeline produces binaries for these targets:

    aarch64-apple-darwin
    x86_64-apple-darwin
    x86_64-unknown-linux-gnu
    x86_64-unknown-linux-musl
    aarch64-unknown-linux-gnu
    x86_64-pc-windows-msvc
    aarch64-pc-windows-msvc

All binaries are statically linked where possible. The Linux musl
target exists for users on Alpine and similar systems.

### 5.3 Release engineering

- Tooling: `cargo-dist` orchestrates the release pipeline.
- CI: GitHub Actions, triggered on tag push.
- Channels:
  - GitHub Releases (primary).
  - Homebrew tap at `memryzed/homebrew-tap`.
  - Scoop bucket at `memryzed/scoop-bucket`.
  - `cargo install memryzed` from crates.io as a fallback.
  - npm wrapper at `@memryzed/cli` is a stretch goal for v1.

### 5.4 Hosting

The install scripts and the marketing site at `memryzed.com` are
hosted on Cloudflare Pages for v1. Migration to AWS (S3 plus
CloudFront) is planned alongside the future cloud product.

## 6. Data location and privacy

### 6.1 Data directory

All Memryzed data lives under a single directory per user:

    Unix:    ~/.memryzed/
    Windows: %LOCALAPPDATA%\memryzed\

The path can be overridden via `--data-dir`, the
`MEMRYZED_DATA_DIR` environment variable, or `[general] data_dir` in
`config.toml`.

The directory contains the binary, the database, the embedding
model, the configuration, and the audit log. It is the only place
Memryzed writes outside of operating-system temporary directories
during transient operations.

### 6.2 Network behavior

By default, Memryzed makes network calls only:

- During `memryzed init`, to download the embedding model from the
  configured model host.
- During `memryzed update` or the once-a-day startup check (if not
  disabled), to query GitHub Releases.

It does not phone home, send telemetry, or transmit memories to any
server.

### 6.3 Telemetry

Off by default. When opted in via `[telemetry] enabled = true`, the
client posts a small counters payload. The payload contains no user
content, no queries, no filenames, no project names. The exact
payload shape is documented in `docs/architecture.md`.

## 7. Memory model

See `docs/concepts.md` for the user-level explanation. The
specification:

### 7.1 Scopes

Three scopes, each represented by `(scope_kind, scope_id)`:

    global   scope_id = NULL
    project  scope_id = a project ID (proj_* or proj_local_*)
    session  scope_id = a session ID (sess_*)

A memory belongs to exactly one scope. There is no inheritance; a
fact in `project` scope is not also in `global` scope.

### 7.2 Kinds

Four kinds of memory:

    preference   How the user likes things done.
    fact         An objective statement about the user, project, or world.
    decision     A choice made during work, often with reasoning.
    todo         A pending task or follow-up.

The kind affects how the agent should treat the memory but is not
used in retrieval ranking.

### 7.3 Status

Four statuses:

    pending    Proposed by the extractor; awaiting user review.
    approved   Active and used in retrieval.
    pinned     Active and never expires.
    archived   Excluded from retrieval; kept for audit.

Hard deletion is supported via `memryzed forget <id> --hard` but is
opt-in.

### 7.4 Confidence and auto-approval

The extractor assigns a confidence score from 0.0 to 1.0 to each
proposed memory. If `confidence >= [memory] auto_approve_threshold`
(default 0.85), the memory is auto-approved. Otherwise it goes to
the pending queue. Direct user input via `memryzed remember` or via
the agent's `remember` tool always becomes approved.

### 7.5 Expiration

A memory may have an `expires_at` timestamp. Expired memories are
treated as archived and excluded from retrieval. A daily background
sweep applies the transition. Pinned memories never expire,
regardless of `expires_at`.

## 8. Retrieval

### 8.1 Hybrid score

For a query string Q against a candidate memory M:

    score(Q, M) = w_v * vector_sim(Q, M) +
                  w_f * fts_score(Q, M) +
                  w_r * recency_boost(M)

Where:

- `vector_sim` is cosine similarity between the query embedding and
  the memory embedding.
- `fts_score` is the BM25 score from FTS5, normalized to [0, 1].
- `recency_boost` is `exp(-age_in_days / 30)`, capped at 1.0.
- `w_v`, `w_f`, `w_r` come from `config.toml` (defaults 0.6, 0.3,
  0.1).

Pinned memories receive a flat additive bonus of 0.1, capped so the
final score does not exceed 1.0.

### 8.2 Filtering

Before scoring:

- Memories with `status NOT IN ('approved', 'pinned')` are excluded.
- Memories with `expires_at < now` are excluded.
- If the query specifies a scope, only memories in that scope are
  considered.

After scoring, the top N memories are returned, where N is the
client-supplied `limit` capped at `[retrieval] max_results`.

### 8.3 Embedding model

v1 ships BGE-small-en-v1.5 (384-dim) via `fastembed-rs` running
ONNX. It is downloaded on first `init`. Switching models is allowed
in code but requires regenerating all embeddings; v1 does not
expose model selection in the user-facing config.

## 9. Sessions

### 9.1 Definition

A session is a per-task working state record scoped to a project.
A session has:

- An ID, `sess_<12 hex>`.
- A project ID.
- A title.
- A state blob (JSON, opaque to Memryzed).
- A status (`active`, `paused`, `completed`, `archived`).
- Timestamps and a pinned flag.

### 9.2 Lifecycle

- A session is created by the first call to `checkpoint` for a
  project that has no active session.
- Subsequent `checkpoint` calls update the active session in place.
- `resume` returns the most recent session for the current project,
  or a session matching the supplied ID.
- `end_session` marks a session as completed and archives it.
- An `active` session that has not been touched in 24 hours
  transitions to `paused` automatically. A `paused` session is
  fully resumable.
- Sessions with status other than `pinned` are archived after 30
  days of inactivity. Archived sessions are not deleted; they fall
  out of normal listings but can be retrieved by ID.

### 9.3 State blob conventions

The state blob is opaque to Memryzed but to support cross-agent
resumption, agents are encouraged to use these keys:

    open_files       List of file paths.
    cwd              Working directory absolute path.
    recent_turns     List of {role, content} objects.
    last_commands    List of recent shell commands or tool calls.
    task_summary     Human-readable description of the current task.

Agents may include additional keys. Unknown keys are ignored by
Memryzed and by other agents.

### 9.4 Project identity

A project ID is computed from the working directory:

1. If `git config --get remote.origin.url` returns a URL, normalize
   it (strip credentials, lowercase host, remove trailing `.git`),
   compute SHA-256, take the first 12 hex characters, prefix with
   `proj_`.
2. Otherwise, hash the absolute path of the working directory and
   prefix with `proj_local_`.

Each `serve` start records the working directory in the project's
`local_paths` JSON array. This unifies clones of the same repo at
different paths.

## 10. MCP surface

The MCP surface is documented in `docs/mcp-reference.md`. The
specification:

### 10.1 Tools

Eight tools, in order of expected call frequency:

    recall            Read.
    remember          Write a single memory.
    list_memories     Read for transparency.
    forget            Archive.
    checkpoint        Write session state.
    resume            Read session state.
    list_sessions     Read.
    end_session       Mark complete.

The set is closed for v1. Adding tools requires a major version bump
in v2 unless they are obvious extensions of existing tools.

### 10.2 Server identity

The handshake reports:

    name:     "memryzed"
    version:  the binary version
    protocol: the MCP protocol version supported

### 10.3 Errors

All errors map to one of the documented error codes. The full set
in v1:

    not_found           A referenced ID does not exist.
    invalid_scope       The scope value is not recognized.
    invalid_argument    A parameter is malformed or missing.
    storage_error       A database or filesystem error occurred.
    not_initialized     The data directory has not been initialized.
    rate_limited        The client exceeded a per-minute limit.

Error responses always include a human-readable message.

### 10.4 Rate limits

Defaults per client per minute:

    recall:         60
    remember:       30
    other tools:    60

Limits use a sliding window. Configurable via
`[server] rate_limit_*` in `config.toml`.

### 10.5 Pre-init behavior

If `memryzed serve` is invoked and the data directory has not been
initialized:

- Initialize silently with defaults.
- Log the auto-init in the audit log.
- Continue serving.

If invoked as a CLI command (anything other than `serve`), prompt
the user with the same content as `memryzed init` would show.

## 12. The TUI for review

`memryzed review` opens a full-screen TUI built with `ratatui`.

### 12.1 Layout

    Top bar:    Memryzed - Pending memories (<count>)
    List:       One row per pending memory, scrollable.
    Detail:     Full content, source, scope, confidence.
    Footer:     Keybindings.

### 12.2 Keybindings

    Up / Down    Move selection.
    a            Approve selected.
    e            Edit selected (opens $EDITOR).
    r            Reject selected (archives).
    p            Approve and pin.
    A            Approve all in view.
    /            Search filter.
    q            Quit.

### 12.3 Behavior

- Approving a memory transitions its status to `approved` and
  triggers embedding generation if not already done.
- Rejecting archives the memory.
- Edits update content and re-embed.
- Search filters the visible list by full-text match against
  content.

## 13. Extractor

### 13.1 Rule-based patterns

The rule-based extractor matches the following patterns in user
messages observed by the agent. Each pattern produces a candidate
memory with the noted confidence range.

Direct request to remember (confidence 1.0, auto-approve):

    "remember (that )?(.+)"
    "don't forget (that )?(.+)"
    "make a note (that )?(.+)"

Stated preference (confidence 0.85 to 0.95):

    "I (always|usually|prefer) (use|do|write) (.+)"
    "I prefer (.+) over (.+)"
    "I like (to use|using) (.+)"

Project fact (confidence 0.85 to 0.95):

    "this (project|repo|repository|codebase) uses (.+)"
    "the (deploy|build|test|lint) command is (.+)"
    "(.+) is the owner of (.+)"

Correction (confidence 0.95, updates existing fact):

    "actually (it's|its|it is) (.+) not (.+)"
    "no, (.+) instead of (.+)"

The extractor passes content through normalization (lowercasing only
proper nouns are excluded, trimming, removing trailing punctuation)
before storing.

### 13.2 Optional Ollama extractor

When `[extractor] ollama_enabled = true`:

- The extractor sends the most recent N turns to the configured
  Ollama model.
- The prompt asks the model to produce zero or more JSON objects
  describing facts to remember, including a confidence score and a
  scope suggestion.
- The output is parsed; well-formed objects become candidate
  memories. Malformed output is logged and discarded.
- If Ollama is unreachable, fall back to rule-based extraction and
  log the failure.

### 13.3 Where extraction observes content

The v1 extractor only observes content that the agent passes to
Memryzed via the `remember` tool or via a future `extract_from`
tool. v1 does not have a passive transcript hook.

This is conservative on purpose. The product can be expanded later
once the rule-based extraction is shown to be useful and trusted.


## 14. Storage

The full schema is in `docs/data-model.md`. The specification:

### 14.1 Engine

- SQLite, WAL mode, busy timeout 5000 ms.
- Extensions: `sqlite-vec` (loaded as a runtime extension), FTS5
  (built-in).
- Single database file at `<data_dir>/db.sqlite`.

### 14.2 Tables

The v1 schema includes:

    memories         Primary table for facts.
    memory_vectors   Virtual table backed by sqlite-vec.
    memory_fts       Virtual table backed by FTS5.
    projects         Repository identities.
    sessions         Session state per project.
    recall_log       Structured event log.
    meta             Single-row metadata table.

### 14.3 Migrations

Schema versioning uses `PRAGMA user_version`. Migration files live
at `crates/memryzed-core/migrations/NNN_description.sql`. The runner
applies all pending migrations on every server start, in
transactions.

For v1, migrations should remain backward-compatible at the schema
level so an older binary can open a newer database for at least one
minor version. Destructive migrations (column drops, type changes)
require a major version bump.

### 14.4 Audit log

Append-only JSONL at `<data_dir>/audit.log`. Rotated when the file
exceeds 10 MB, keeping rotated files for 90 days.

### 14.5 Backups

`memryzed export` is the primary backup mechanism. The format is
versioned and portable. Direct file copy of `db.sqlite` (with WAL
and SHM files) while Memryzed is not running is also supported.

## 15. Integrations with MCP clients

### 15.1 Detection

`memryzed install` looks for these client-specific config files:

    Claude Code:    ~/.claude/mcp.json
    Kiro CLI:       ~/.kiro/settings/mcp.json
    Cursor:         ~/.cursor/mcp.json
    Codex CLI:      ~/.codex/mcp.json
    Continue:       ~/.continue/config.json
    Copilot CLI:    determined by the gh-copilot extension layout

For each file found, the user is prompted to confirm before any
modification.

### 15.2 Config block written

The exact JSON written depends on the client's config schema, but
the substance is:

    {
      "mcpServers": {
        "memryzed": {
          "command": "<absolute path to memryzed>",
          "args": ["serve"]
        }
      }
    }

For clients that use environment variables to scope the working
directory, the appropriate value is added.

### 15.3 Backups

Every modified config file is backed up to `<file>.memryzed.bak`
before writing. `memryzed install --print` prints the block without
modifying anything.

### 15.4 Removal

`memryzed uninstall --unwire` removes the `memryzed` entry from each
configured client.

## 16. Updates and self-update

### 16.1 Check on startup

If `[updates] check_on_startup = true`, the server checks
`https://api.github.com/repos/memryzed/memryzed/releases/latest` at
most once per day. The check runs in a background task and never
blocks startup.

If a newer release is found, the server prints a notice on stderr
and writes an `update_available` event to the audit log. The notice
includes the version and a one-line summary linking to the release
notes.

### 16.2 Manual update

`memryzed update` performs the install:

1. Fetches the matching binary from the latest release.
2. Verifies its checksum against the release manifest.
3. Atomically replaces the running binary using a tempfile and
   rename, or via the `self_update` crate where supported.
4. Prints the new version.

### 16.3 Auto-install

`[updates] auto_install` is reserved. In v1 it is documented but
ignored. Auto-install will not ship in the v1.0 release.

## 17. Observability

### 17.1 Log levels

Standard tracing levels: trace, debug, info, warn, error. Default
is info. Configurable via `MEMRYZED_LOG_LEVEL` or `[general]
log_level`.

### 17.2 Audit trail

Every state-changing operation writes to the audit log. See
`docs/data-model.md` for the kinds. The audit log is the
authoritative trust surface; it never depends on agent or client
cooperation.

### 17.3 `memryzed doctor`

The doctor command runs these checks and prints a result for each:

- Binary location is in PATH.
- Data directory exists and is writable.
- Database opens cleanly and `PRAGMA integrity_check` returns `ok`.
- Embedding model files are present and readable.
- Each known MCP client config either contains a Memryzed entry or
  is absent.
- Recent activity is flowing (within the last 24 hours, if any).

Exit zero on full success, non-zero on any failure.

## 18. Performance targets

These are the v1.0 targets, measured on a typical developer laptop
(2024-era M-series Mac or x86_64 Linux laptop) with a 10,000-memory
database:

- `recall` p50 latency: under 30 ms.
- `recall` p99 latency: under 100 ms.
- `remember` p50 latency: under 20 ms (excluding embedding compute).
- Embedding compute for one memory: under 100 ms.
- Cold-start latency for `memryzed serve`: under 200 ms before
  ready.
- Memory footprint of a running `serve` with the model loaded:
  under 250 MB resident.

These are targets, not contracts. CI runs benchmarks against them
and flags regressions.

### 18.1 Quality benchmarks

Latency targets above measure speed. Quality benchmarks measure
retrieval correctness and are governed separately by
`docs/specs/benchmarks.md`. The v1.0 release ships without
published quality numbers; those are produced against the v1.0
implementation and published with v1.1 or v1.2. We do not gate
v1.0 on hitting any specific recall number, but we do gate the
first quality publication on numbers we are willing to defend in
public.

## 20. Testing

See `docs/development/testing.md` for the full testing strategy. The
specification:

- Unit tests on every module in `memryzed-core`.
- Integration tests that spin up an in-process server and exercise
  the MCP surface end to end against a real SQLite database.
- A small fixture set of sample conversations and expected
  extractor output.
- A migration test that loads each historical schema version and
  applies migrations to current.
- Smoke tests run on every supported target in CI.

CI gates:

- `cargo test` passes on Linux, macOS, and Windows.
- `cargo clippy` passes with no warnings on the default profile.
- `cargo fmt` is clean.
- `cargo deny check` passes.

## 21. Acceptance criteria for v1.0

The release ships when all of the following are true:

1. The install commands documented in section 5.1 work on every
   build target listed in section 5.2.
2. After install and `memryzed init`, the binary passes
   `memryzed doctor` on a clean machine.
3. `memryzed install` correctly auto-detects and writes
   configuration for at least Claude Code, Kiro CLI, and Cursor.
4. The eight MCP tools work end to end against at least one real
   client.
5. The CLI surface in section 11 is fully implemented and
   documented.
6. The TUI review interface is functional with the keybindings in
   section 12.2.
7. The performance targets in section 18 hold on the reference
   laptop.
8. `memryzed export` produces a file that `memryzed import`
   restores with no data loss for memories, sessions, and
   projects.
9. The full documentation set in `docs/` is current.
10. CHANGELOG entries exist for every shipped feature under
    `[Unreleased]` and are moved to a v1.0.0 section on tag.
11. The release pipeline produces signed-checksum binaries for all
    supported targets and they are downloadable through every
    supported install channel.
